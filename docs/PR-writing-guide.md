# PR 작성 톤 가이드

`.github/pull_request_template.md`를 채울 때의 문체 규약. pr-create 스킬이 이 가이드를 따른다.

## 원칙

1. **한 줄 요약 먼저.** PR이 무엇을 하는지 한 문장으로. 제목과 중복돼도 괜찮다.
2. **"무엇을 / 왜 / 어떻게 쓰는지" 순서.** 변경 사항은 프레임워크·API 이름 나열로 끝내지 않는다. 한 줄 풀이를 붙인다.
   - 나쁜 예: `- SwiftData, Keychain 적용`
   - 좋은 예: `- 계정 메타는 SwiftData(@Model)로, 비밀번호는 Keychain으로 분리 저장(방식 B) — 평문이 DB에 남지 않게 하려고.`
3. **범용 명령 블록 금지.** `xcodegen generate` / `xcodebuild` 같은 누구나 아는 절차는 "직접 확인하는 법"에 넣지 않는다. 이 변경에만 필요한 특수 절차(예: 특정 설정 토글, 시드 데이터 주입)가 있을 때만 적는다.
4. **자동 리뷰·검증 결과는 표로.** 빌드/SwiftLint/테스트 결과는 표나 체크박스로 한눈에.
5. **UI 변경은 스크린샷 Before/After 표.** 시뮬레이터 캡처. 라이트 모드 기준(v1).

## 보안 앱 특칙

- PR 본문·스크린샷·로그에 **시크릿 평문·실제 비밀번호 값을 넣지 않는다**(로컬 전용 보안 앱). 더미 값을 쓴다.
- 시크릿을 다루는 변경이면 "방식 B 위반 없음(평문 미저장)"을 명시한다.

## 분기

- base는 **`develop`**(통합). `main`은 배포 전용 — 릴리스 때만 올린다.
- 글로벌 규칙: 커밋·PR 본문에 `Co-Authored-By:`나 "🤖 Generated with Claude Code"를 넣지 않는다.
