## 2. 제품 개요

### 2.1 한 줄 요약

내 기기가 아닌 환경이거나 자동완성이 동작하지 않는 상황에서, 저장해둔 아이디·비밀번호를 **빠르게 찾아 바로 입력**하기 위한 로컬 전용 경량 iOS 비밀번호 관리 앱임.

### 2.2 배경 및 문제 정의

현재 본인은 자주 쓰는 계정 정보를 **메모장 앱 하나에 전부 평문으로 저장**해두고 필요할 때마다 찾아 쓰고 있음. 이 방식은 다음 상황에서 특히 불편함.

- **타 기기 로그인**: PC방 등 내 컴퓨터가 아닌 환경에서는 브라우저 비밀번호 자동완성이 없어서 직접 타이핑해야 함
- **자동저장 미동작 사이트**: 일부 사이트는 브라우저 비밀번호 저장 기능이 동작하지 않아 매번 수동 입력해야 함
- **모바일 앱 수동 로그인**: 일부 모바일 앱은 자동 로그인이 안 되어 항상 아이디·비밀번호를 직접 입력해야 함

그런데 메모장에 저장된 계정 수가 많아지면서 **원하는 계정을 찾는 것 자체가 번거로운 작업**이 됨. 메모장은 검색·구조화가 약하고, 평문으로 저장되어 보안상으로도 취약함. 또한 계정마다 2차 비밀번호처럼 추가로 기억해야 할 값이 있는 경우 메모장에서는 구조적으로 관리하기 어려움.

### 2.3 제품 목표

- 저장된 계정을 **1~2초 내에 검색**해서 찾을 수 있게 함
- 찾은 값을 **필드별로 한 번의 탭으로 복사**하거나 화면에서 즉시 확인할 수 있게 함
- 계정마다 **필요한 필드를 자유롭게 추가**할 수 있게 해, 2차 비밀번호 등 다양한 항목을 함께 보관함
- 메모장보다 **안전하게**(로컬 암호화 저장) 보관함
- 앱 실행과 검색이 즉각 반응하는 **가볍고 빠른 경험**을 제공함

### 2.4 비목표 (Non-Goals) — v1 기준

명확히 v1 범위에서 제외하는 것들임. "빠르고 가벼움"을 지키기 위한 의도적 배제임.

- 멀티 디바이스 클라우드 동기화 (iCloud 등)
- 팀·가족 비밀번호 공유
- 비밀번호 유출 검사·강도 분석 같은 부가 보안 기능
- Android·웹 버전
- 브라우저 확장 자동입력
- 다크 모드 (추후 v1.1로 이관)

### 2.5 보안 모델 (전제)

본 앱의 보안 설계는 다음 전제 위에 세워짐.

- **모든 데이터는 기기 로컬에만 저장**되며 네트워크로 전송하지 않음. 서버·클라우드가 없으므로 원격 유출 경로가 존재하지 않음
- 따라서 데이터 노출의 사실상 유일한 경로는 **타인이 본인 휴대폰에 물리적으로 접근하는 경우**로 한정됨
- 이 물리적 접근에 대한 **1차 방어선은 iOS 기기 자체의 잠금**(패스코드 / Face ID / Touch ID)임. 기기가 잠겨 있으면 앱과 데이터 모두 접근 불가임
- **앱 레벨의 추가 잠금은 그 위에 선택적으로 얹는 2차 보호막**임. 폰을 잠깐 빌려주거나 잠금이 풀린 상태로 두는 경우를 대비하는 용도이며, 강제하지 않고 **사용자가 켜고 끌 수 있는 선택 기능**으로 제공함
- 어깨너머 엿보기(shoulder surfing)에 대비해 **모든 시크릿 필드는 기본 마스킹(블러)** 되며, 필요한 필드만 터치해서 드러냄

> 정리하면, **데이터 저장 자체의 보호(Keychain 암호화)** 와 **앱 진입 잠금(생체/암호)** 은 별개 레이어임. 저장 보호는 항상 적용되고(메모장 평문 대비 핵심 개선점), 진입 잠금만 선택임.

## 3. 타겟 사용자

주 사용자는 **본인(1인)** 이며, 개발자인 동시에 실사용자임. 따라서 별도의 온보딩·튜토리얼·범용 UX 고려 부담이 적고, 본인의 실제 사용 패턴에 최적화하면 됨.

핵심 사용 맥락은 "급하게, 다른 화면을 보면서, 빠르게 계정 정보를 꺼내 쓰는 상황"임. 즉 **조회 속도와 검색 편의가 최우선**이며, 진입 잠금을 기본값으로 강제하지 않는 것도 이 빠른 접근성을 위한 선택임.

## 4. 핵심 사용 시나리오 (User Stories)

| ID | 시나리오 | 사용자 스토리 |
| --- | --- | --- |
| US-1 | PC방 로그인 | 다른 컴퓨터에서 로그인할 때, 폰에서 앱을 열어 해당 사이트 계정을 빠르게 찾아 보고 직접 입력함 |
| US-2 | 자동저장 안 되는 사이트 | 자동완성이 안 뜨는 사이트에서, 앱에서 비밀번호를 확인해 입력함 |
| US-3 | 모바일 앱 수동 로그인 | 자동 로그인이 안 되는 앱을 쓸 때, 마이아카이브에서 비밀번호를 복사해 앱에 붙여넣음 |
| US-4 | 계정 추가 | 새 계정을 만들면 서비스명·아이디·비밀번호를 빠르게 등록하고, 원하면 식별용 색을 지정함 |
| US-5 | 추가 필드 보관 | 2차 비밀번호·보안 질문·계좌 PIN 등 계정마다 다른 항목을, 필드를 직접 추가해 함께 보관함 |
| US-6 | 선택적 앱 잠금 | (원하는 경우) 설정에서 앱 잠금을 켜두어, 폰을 타인이 만지더라도 계정 정보가 노출되지 않게 함 |

US-1, US-2는 **다른 기기를 보면서 폰 화면을 읽어 타이핑**하는 상황이라 "빠른 검색 + 마스킹 해제"가 중요함.
US-3는 **같은 폰 안에서 앱 전환**이라 "필드별 빠른 복사"가 중요함.

## 5. 기능 요구사항 (MoSCoW)

### 5.1 Must Have (MVP 필수)

| ID | 기능 | 설명 |
| --- | --- | --- |
| F-1 | 계정 CRUD + **사용자 정의 필드** | 계정 추가·조회·수정·삭제. 기본 필드: 서비스명, 아이디, 비밀번호, 메모(선택), URL(선택). 여기에 더해 사용자가 **이름과 값을 가진 필드를 직접 추가·삭제**할 수 있음 (예: "2차 비밀번호", "보안 질문 답"). 추가 필드 개수에 제한 없음. **추가/수정 화면은 ‘기본 정보(서비스명) · 색상 · 필드(아이디·비밀번호·커스텀) · 추가 정보(URL·메모) · 즐겨찾기’ 섹션으로 구성**(Design.md 2.4) |
| F-2 | 실시간 검색 | 서비스명·아이디 기준 부분 일치 검색. 입력과 동시에 목록 필터링 |
| F-3 | **터치 기반 마스킹 해제** | 상세 화면의 **시크릿 필드 값(아이디·비밀번호·커스텀)은 기본 마스킹(블러 처리)**. 마스킹된 값을 **직접 터치하면 그 필드만 평문으로 드러남**. 표시/숨김 토글이 아니라, 터치 시 드러나는 단방향 동작임. 다시 가리는 것은 상세 화면을 벗어났다 재진입하면(또는 약 22초 경과 후) 자동 재마스킹으로 처리함. URL·메모는 비시크릿(항상 표시) |
| F-4 | **필드별 개별 복사** | 각 필드마다 **독립된 복사 버튼**을 둠. 아이디·비밀번호·커스텀 필드가 **한 번에 묶여 복사되지 않으며**, 사용자가 원하는 필드만 개별 복사함. 복사 시 토스트 피드백 + 버튼 체크 표시 |
| F-5 | 선택적 앱 잠금 | 설정에서 켤 수 있는 생체 인증(Face ID/Touch ID) 또는 기기 암호 잠금. **기본값은 꺼짐**이며, 켜면 앱 진입 시 인증을 요구함. 끄면 바로 실행됨. 설정에 ‘지금 잠그기’ 제공 |
| F-6 | 암호화 로컬 저장 | 비밀번호 및 민감 필드 값을 평문이 아닌 안전한 형태(Keychain)로 로컬 저장. **잠금 설정과 무관하게 항상 적용됨** |
| F-7 | 설정 화면 | 앱 잠금 on/off, 클립보드 만료 시간, 정렬 방식, 정보(버전·저장 방식·네트워크)를 담음 |

> F-5와 F-6의 관계: F-6은 "저장된 데이터 자체를 OS가 암호화·보호"하는 항상 켜진 레이어임. F-5는 "앱을 여는 순간 사람을 인증"하는 선택 레이어임. 잠금을 꺼도 비밀번호가 평문으로 저장되는 일은 없음.
>
> F-3과 F-4의 관계: F-3은 화면에서 값을 **보는** 방식(마스킹/터치 해제), F-4는 값을 **가져가는** 방식(필드별 복사)임. 둘 다 "필요한 값만, 안전하게"라는 원칙을 따름.

### 5.2 Should Have

| ID | 기능 | 설명 |
| --- | --- | --- |
| F-8 | 클립보드 자동 삭제 | 복사 후 일정 시간(설정: 30 / 60 / 120초, 기본 60) 뒤 클립보드에서 자동 제거 |

## 6. 비기능 요구사항

### 6.1 성능 (가장 중요한 축)

- 콜드 스타트 후 첫 목록 화면 표시까지 **1초 이내**를 목표로 함 (앱 잠금이 꺼져 있으면 인증 단계가 없어 더 빠름)
- 검색 입력 시 **체감 지연 없이** 즉시 필터링됨
- 계정 수는 많아야 수백 개 수준이므로, 앱 실행 시 **메타데이터를 메모리에 로드해 인메모리 검색**하는 전략으로 즉각 반응성을 확보함 (민감 값은 표시 시점에만 Keychain에서 조회)

### 6.2 보안

- 비밀번호·민감 필드 값 **평문 저장 금지**. Keychain 기반 보관 (항상 적용)
- 앱 진입 잠금(생체/암호)은 **선택 기능**으로 제공. 사용자가 설정에서 켜고 끌 수 있음
- 모든 시크릿 필드는 기본 마스킹(블러), 터치 시에만 해제
- 복사한 값은 **클립보드 만료** 적용
- **앱 잠금이 켜진 경우**, 멀티태스킹 스냅샷에서 민감 정보가 노출되지 않도록 화면을 가림
- 스크린샷·화면 녹화에 대해 iOS 한계 내에서 가능한 대응을 적용 (7.7 참고)

### 6.3 가벼움 / 의존성

- 외부 SDK·네트워크 의존성 **최소화(이상적으로 0)**
- **완전 오프라인** 동작 (네트워크 권한 불필요)
- 앱 용량을 작게 유지

### 6.4 사용성 / 디자인

- **검색 중심** 단일 화면 구조, 한 손 조작 고려
- 빈 상태·에러 상태를 명확히 처리(빈 계정/검색 0건 전용 화면 포함)
- 잠금을 켜지 않은 사용자에게는 **불필요한 인증 단계 없이** 즉시 사용 가능한 흐름 제공
- 커스텀 필드 추가·삭제는 직관적인 인라인 편집으로 제공
- **라이트 모드 전용**(v1). 비주얼 아이덴티티·토큰은 `Design.md` 1장 참조
- 모든 화면은 iOS HIG 기반 + 약간의 브랜드 커스텀(인디고 액센트)

## 7. 보안 설계

비밀번호 앱의 핵심이므로 별도 항목으로 정리함. "빠르고 가벼움"과 보안을 동시에 만족시키기 위해 **자체 암호화 구현을 최소화하고 검증된 OS API(Keychain)에 의존**하는 방향으로 설계함.

### 7.1 저장 아키텍처 비교 (참고)

| 방식 | 장점 | 단점 |
| --- | --- | --- |
| A. Keychain 단독 | OS가 암호화·보호, 구현 단순 | 정렬·검색용 메타데이터 관리가 번거로움 |
| **B. SwiftData(메타) + Keychain(시크릿)** | 검색·정렬 빠르고 유연, 시크릿은 안전하게 분리 | 두 저장소 동기화 로직 필요 |
| C. 전체 암호화 파일 + 마스터키 | 단일 파일로 관리 | 키 관리·암호화 직접 구현, 실수 위험 |

### 7.2 확정안: 방식 B

- **메타데이터**(서비스명, 아이디, 메모, URL, 즐겨찾기, **색상**, 생성·수정 날짜, 커스텀 필드의 이름·순서·Keychain 참조)는 SwiftData로 저장 → 빠른 검색·정렬
- **시크릿 값**(비밀번호, 커스텀 필드의 실제 값)은 Keychain에 저장하고, SwiftData에는 **Keychain 조회용 키만** 보관 → 시크릿 평문은 영속 DB에 절대 들어가지 않음
- 학습 가치도 큼(SwiftData·Keychain·LocalAuthentication을 모두 다루게 됨)

> 아이디(username)는 검색 키이므로 빠른 검색을 위해 메타데이터에 보관함. 단 화면에서는 다른 시크릿 필드와 동일하게 마스킹되며, 터치 시에만 드러남. (저장 위치와 화면 표시 방식은 별개 레이어임)

### 7.3 데이터 저장 보호 (항상 적용)

- 시크릿은 Keychain에 저장하며, 접근성 속성을 `kSecAttrAccessibleWhenUnlocked` 수준으로 설정해 **기기가 잠금 해제된 상태에서만** 접근 가능하게 함
- 이로써 앱 자체 잠금을 켜지 않아도, **기기가 잠겨 있는 동안에는 데이터가 보호**됨

### 7.4 앱 진입 잠금 (선택 기능)

- 기본값은 **꺼짐**. 설정(F-7)에서 사용자가 켤 수 있음
- 켜면 `LocalAuthentication`의 `LAContext`로 Face ID / Touch ID 인증을 요구하고, 생체 인증 불가 시 기기 암호(passcode)로 폴백함
- 앱 진입 시 1회 인증, 백그라운드 후 재진입 시 재인증 (잠금이 켜진 경우에 한함). 설정에서 ‘지금 잠그기’로 수동 잠금 가능

### 7.5 필드 마스킹 (항상 적용)

- 상세 화면의 시크릿 필드 값(아이디·비밀번호·커스텀)은 기본 마스킹(블러)됨
- 마스킹된 값을 직접 터치하면 해당 필드만 평문으로 드러남 (단방향, 토글 아님)
- 화면 재진입 또는 약 22초 경과 시 자동 재마스킹. URL·메모는 비시크릿

### 7.6 클립보드 보호

- `UIPasteboard`에 복사 시 **만료 시간(expirationDate)** 설정으로 일정 시간(30/60/120초) 뒤 자동 제거
- 필드별로 개별 복사하며, 복사 시 사용자에게 토스트 피드백 제공

### 7.7 스크린샷·화면 녹화 대응 (가능 범위)

iOS는 안드로이드의 `FLAG_SECURE`처럼 **스크린샷을 원천 차단하는 공식 API를 제공하지 않음**. 따라서 완벽한 차단은 불가능하며, 다음의 가능한 대응을 조합해 위험을 완화함.

| 대응 | 방법 | 한계 |
| --- | --- | --- |
| 화면 녹화·미러링 감지 | `UIScreen.isCaptured` 관찰 → 캡처 중이면 민감 화면을 가림 | 녹화/미러링에만 해당 |
| 스크린샷 사후 감지 | `userDidTakeScreenshotNotification` → 찍힌 직후 경고·로깅 | 이미 찍힌 뒤라 사전 차단은 아님 |
| 보안 입력 레이어 트릭(선택) | `UITextField`의 secure entry 레이어에 민감 뷰를 올려 캡처물에서 가려지게 함 | 비공식 기법, OS 업데이트로 동작이 바뀔 수 있음. 적용 여부는 구현 단계에서 판단 |

> 결론: "가능하면 막는다"는 목표는 위 조합으로 **부분적으로만** 달성 가능함. 이를 보완하는 1차 방어선은 결국 보안 모델(로컬 저장 + 기기 잠금 + 필드 마스킹)임을 PRD에 명시함.

## 9. 데이터 모델 (방식 B)

시크릿 값은 영속 DB에 저장하지 않고 Keychain 참조 키만 보관함.

### 9.1 Credential (SwiftData `@Model`) — 메타데이터

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| id | UUID | 고유 식별자 |
| serviceName | String | 서비스·사이트명 (검색 대상) |
| username | String | 아이디 (검색 대상, 화면에서는 마스킹) |
| passwordRef | String | 비밀번호의 Keychain 조회 키 |
| memo | String? | 메모 (선택) |
| urlString | String? | URL (선택) |
| **colorHex** | **String** | **계정 아바타 색 (F-15). 사용자가 고른 HEX, 미지정 시 서비스명 해시로 자동 배정** |
| customFields | [CustomField] | 사용자 정의 필드 목록 (`@Relationship`) |
| isFavorite | Bool | 즐겨찾기 여부 (정렬 1순위) |
| createdAt | Date | 생성 시각 (수정 이력 없을 때 표시·정렬 기준) |
| updatedAt | Date? | 수정 시각 (있을 때 표시·정렬 기준) |

> **변경점(v0.5)**: 기존 `lastUsedAt`(마지막 사용)을 화면 표시·정렬에서 제거하고 **`updatedAt`(없으면 `createdAt`)** 으로 일원화함. 목록/상세 보조 텍스트는 “수정/생성” 표기. 신규 저장 시 `createdAt`=현재, `updatedAt`=nil. 편집 저장 시 `updatedAt`=현재. `colorHex` 필드 신규 추가.

### 9.2 CustomField (SwiftData `@Model`) — 사용자 정의 필드 구조

| 필드 | 타입 | 설명 |
| --- | --- | --- |
| id | UUID | 고유 식별자 |
| label | String | 필드 이름 (예: "2차 비밀번호") |
| valueRef | String | 실제 값의 Keychain 조회 키 |
| sortOrder | Int | 상세 화면 내 필드 표시 순서 |

> 커스텀 필드의 **값 자체는 Keychain**에, **이름·순서·참조 키는 SwiftData**에 둠. 즉 어떤 필드를 만들었는지는 메타에서 알 수 있지만, 값은 Keychain에서만 꺼낼 수 있음.

### 9.3 Keychain 항목

| 키 | 값 |
| --- | --- |
| passwordRef (= 계정 id 기반) | 비밀번호 값 |
| valueRef (= 커스텀 필드 id 기반) | 커스텀 필드 값 |

### 9.4 앱 설정 (UserDefaults)

| 키 | 타입 | 기본값 | 설명 |
| --- | --- | --- | --- |
| isAppLockEnabled | Bool | false | 앱 진입 잠금 사용 여부 |
| clipboardExpirySec | Int | 60 | 클립보드 자동 삭제 시간(초): 30 / 60 / 120 |
| sortMode | String | "favoriteRecent" | 정렬 모드 (기본 / 이름순, F-14) |

> 정렬 로직: `isFavorite == true` 그룹을 먼저, 그 안과 그 아래(일반 그룹)는 각각 **`updatedAt ?? createdAt` 내림차순**. 신규 저장 시 활동시각을 현재로 두면 일반 그룹 최상단에 위치함. 이름순 모드(F-14) 선택 시 즐겨찾기 우선 없이 `serviceName` 오름차순으로 정렬.

### 10.2 핵심 플로우

잠금 설정에 따라 진입 흐름이 갈림.

```
[앱 잠금 OFF — 기본값]
앱 실행 → 메인 목록 (바로 진입)
   → 검색어 입력 → 결과 항목 탭 → 상세 화면
   → 시크릿 값 터치로 마스킹 해제 / 필드별 복사 → (타 기기 입력 / 타 앱 붙여넣기)

[앱 잠금 ON — 선택 시]
앱 실행 → 잠금 화면 → Face ID 인증 → 메인 목록
   → (이하 동일)
```

커스텀 필드 추가 + 색상 지정 플로우:
```
추가/수정 화면 → 서비스명 입력 → (선택) 색상 스와치 지정
   → "필드 추가" → 필드 이름 입력(예: 2차 비밀번호) + 값 입력
   → 저장 시 값은 Keychain, 구조·색상은 SwiftData에 기록
```

빠른 복사(Should, 추후): 목록에서 스와이프 또는 롱프레스로 상세 진입 없이 주요 필드를 바로 복사할 수 있게 함.

| v3.0 | iCloud 동기화, 가져오기/내보내기 |

> 참고: 디자인 프로토타입에는 F-8(클립보드 만료 설정), F-9(즐겨찾기/정렬), F-14(이름순)도 이미 반영되어 있어, v1.0에서 함께 구현해도 무방함. 위 표는 원 로드맵 기준의 최소 분리이며 구현 순서는 마일스톤(12장)을 따름.

> **AutoFill을 v2로 둔 이유**: US-3(모바일 앱 수동 로그인)을 가장 근본적으로 해결하는 건 iOS의 AutoFill 확장이지만, 구현 난이도가 높음. v1은 "수동 복사"로 빠르게 가치를 검증하고, AutoFill은 학습이 무르익은 v2에서 도입하는 게 합리적임.

## 12. 개발 마일스톤 (Swift 학습 연계)

학습과 개발을 묶은 단계별 계획임. 각 단계에서 다룰 핵심 개념을 함께 표기함.

| 단계 | 목표 | 학습 포인트 |
| --- | --- | --- |
| M1 | UI 골격 (더미 데이터로 목록·상세·추가, **커스텀 필드 동적 입력 + 색상 선택 UI**) | SwiftUI 기초, `NavigationStack`, `List`, `@State`, `@Binding`, `ForEach` 동적 행 |
| M2 | 메타데이터 영속화 (Credential + CustomField 관계, color·created/updated 포함) | SwiftData `@Model`, `@Relationship`, `@Query`, `ModelContext` |
| M3 | 시크릿 저장·조회 (비밀번호 + 커스텀 필드 값) | Keychain Services, Security 프레임워크 |
| M4 | 선택적 앱 잠금 + 설정 화면 | LocalAuthentication `LAContext`, UserDefaults, 조건부 화면 분기 |
| M5 | 보안·표시 마감 (블러 마스킹/터치 해제, 필드별 복사+만료, 정렬 규칙(수정/생성), 화면 보호, 스크린샷 대응) | `UIPasteboard`, `ScenePhase`, `UIScreen.isCaptured`, 정렬 로직 |
| M6 | 폴리싱 | 검색 최적화, 빈 상태·에러 처리, 토스트·애니메이션 마감 |

## 13. 리스크 및 고려사항

| 리스크 | 대응 |
| --- | --- |
| 앱 잠금을 끈 상태에서 기기 잠금도 풀려 있으면, 타인이 폰을 만질 때 계정이 노출됨 | 보안 모델상 감수하는 트레이드오프임. 기기 자체 잠금을 1차 방어선으로 신뢰하며, 민감하게 느껴질 경우 설정에서 앱 잠금을 켜면 됨 |
| 스크린샷 원천 차단 불가 (iOS 공식 API 부재) | 화면 녹화·미러링 감지 가림, 스크린샷 사후 감지 경고 등 가능한 대응만 적용. 완벽 차단은 보장하지 않음을 명시 |
| 기기 분실 시 데이터 복구 불가 (v1은 동기화 없음) | v1에서는 "단일 기기 로컬 보관"임을 명확히 인지. 복구는 v3 동기화로 해결 |
| 클립보드를 통한 값 노출 (타 앱이 읽을 수 있음) | 클립보드 만료 + 복사 시 사용자 피드백 |
| 커스텀 필드/시크릿 누락 정합성 (메타는 있는데 Keychain 값이 없는 경우 등) | 필드 삭제 시 메타와 Keychain을 함께 정리. 조회 실패 시 사용자에게 안내 |
| 1인 개발의 보안 검증 한계 | 자체 암호화 대신 검증된 OS API(Keychain) 위주로 구현 |
| iOS 버전 호환 | SwiftData는 iOS 17+. 하위 호환 필요 시 Core Data로 대체 |

## 14. 성공 지표

- 본인이 **실제로 메모장 대신 이 앱을 일상적으로 사용**하게 됨 (핵심 지표)
- 콜드 스타트 → 목록 표시 **1초 이내** (잠금 미사용 시)
- 검색어 입력 후 결과가 **즉시** 반영됨
- 메모장 대비 원하는 계정을 찾는 시간이 **체감상 크게 단축**됨
- 2차 비밀번호 등 추가 항목을 별도 메모 없이 **한 앱에서 관리**하게 됨
