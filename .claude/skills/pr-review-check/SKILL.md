# pr-review-check

리뷰어/팀원의 피드백을 한눈에 정리한다. 이 스킬은 **읽고 분류해 보고만** 한다 — 수정은 하지 않는다. 무엇을 고쳐야 할지 사용자가 판단한 뒤, 실제 수정은 orchestrator/에이전트로 넘긴다.

## 왜

리뷰 코멘트는 inline·리뷰 요약·일반 코멘트로 흩어져 있어 놓치기 쉽다. 한 번에 모아 분류하면 무엇이 블로커이고 무엇이 단순 제안인지 빠르게 판단할 수 있다. 수정과 정리를 분리해 의도치 않은 변경을 막는다.

## 절차

1. **PR 식별** — 번호가 주어지면 그 PR, 아니면 현재 브랜치의 PR(`gh pr view`)을 대상으로 한다.
2. **코멘트 수집**
   - `gh pr view nn --comments` — 일반 코멘트·리뷰 요약.
   - `gh api repos/l2juhan/myArchive-IOS/pulls/nn/comments` — inline 리뷰 코멘트(파일·라인 포함).
   - `gh pr view nn --json reviews` — 승인/변경요청/코멘트 상태.
3. **분류** — 각 코멘트를 아래 셋 중 하나로 분류한다:
   - **수정 필요** — 변경요청·버그·보안·도메인 규칙 위반 지적.
   - **논의 필요** — 설계 이견·트레이드오프·범위 질문(즉답 불가).
   - **반영 완료/단순 확인** — 이미 처리됐거나 칭찬·승인.
4. **보고** — 분류별로 묶어, 코멘트별 `파일:라인 — 요지 — 분류` 형태로 정리한다. 보안/데이터 손실 관련 지적은 맨 위로 올린다.

## 보고 형식

```
## 수정 필요
- KeychainService.swift:42 — 삭제 시 cf_{id} 정리 누락 지적 (도메인 규칙)
## 논의 필요
- 정렬 기본값 favoriteRecent vs name 이견
## 반영 완료/확인
- 토스트 카피 승인됨
```

## 규칙

- 코드를 **고치지 않는다**. 정리·보고가 끝. 수정은 사용자 승인 후 별도 흐름(orchestrator).
- 도메인 규칙(방식 B·마스킹·복사 만료) 위반 지적은 `myarchive-domain` 기준으로 타당성을 메모한다.
- 반복적으로 같은 지적이 나오면 harness-feedback에 남길 것을 제안한다.
