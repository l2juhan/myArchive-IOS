# harness-feedback

하네스 자체를 점진 개선하기 위한 관찰 기록기. 같은 교정이 반복되거나 코드↔문서 drift가 자꾸 생기면, 그건 "사람이 매번 고칠 일"이 아니라 **규칙·훅·스킬·문서가 부족하다는 신호**다. 이 스킬은 그 신호를 잡아 기록한다.

## 왜

반복 교정은 하네스가 비용을 사람에게 떠넘기고 있다는 증거다. 기록해두면 나중에 한 번에 규칙/훅으로 자동화할 수 있다. **작업 흐름을 끊지 않기 위해** 여기서는 고치지 않고 로그에만 남긴다 — 정비는 별도 사이클에서.

## 무엇을 감지하나

- **반복 교정** — 같은 종류의 지적/수정이 두 번 이상(예: 마스킹 초기화 누락, 복사 만료 빠짐).
- **drift** — 코드는 바뀌었는데 문서가 자꾸 뒤처짐, generated 신선도 위반 반복.
- **게이트 마찰** — pre-commit/pre-push가 같은 이유로 반복 실패.
- **스킬 트리거 실패** — 의도한 스킬이 안 불리거나 엉뚱하게 불림.

## 절차

1. **관찰 수집** — 이번 세션(또는 사용자가 가리킨 패턴)에서 반복된 교정·drift를 모은다.
2. **형식 확인** — `docs/design-docs/feedback-log.md`의 기존 형식을 그대로 따른다.
3. **항목 추가** — 파일 맨 아래에 추가(기존 항목 수정 금지):
   ```
   ## [YYYY-MM-DD] 제목
   - 유형: drift | 반복 교정 | health 발견 | 규칙 제안
   - 관찰: 무엇이 반복/문제인가
   - 제안: 규칙/훅/스킬/문서 개선안 (구체적으로 — 어느 파일에 무엇을)
   - 상태: open
   ```
4. **보고** — 무엇을 기록했는지 한 줄로 알리고, 지금 고칠지(별도 사이클) 사용자에게 맡긴다.

## 규칙

- **작업 브랜치에서 규칙/훅/스킬을 직접 고치지 않는다** — 기록만. 정비는 사용자 승인 후 별도로.
- 기존 로그 항목을 덮어쓰지 않는다. 추가만. 처리되면 상태를 `applied`/`wontfix`로 갱신.
- 제안은 실행 가능하게: "어느 훅/스킬/문서에 무엇을 추가/변경"까지 적는다.
- 한 번 일어난 일은 기록 대상이 아니다. **반복**이 기준이다(노이즈 방지).
